MVC MVP MVVM

发表于 2020-11-10 更新于 2021-02-04 分类于 设计模式

MVC

MVC，即模型(model)－视图(view)－控制器(controller)，一种软件设计典范，用一种业务逻辑、数据、界面显示分离的方法组织代码，将业务逻辑聚集到一个部件里面，在改进和个性化定制界面及用户交互的同时，不需要重新编写业务逻辑。

模块分析

Model

提供要展示的数据，包含数据与业务逻辑，提供模型数据查询与模型状态更新等行为，类似于同样提供数据与行为的Javabean。

现代可以更加细分为Value Object（Dao）和业务层（Service），分别掌管数据处理与业务逻辑

类比于最当初调研的GRASP，我觉得model层包含一系列具体处理事务的掌握着信息的信息专家类。

View

负责进行model的展示，是模型中处理数据展示的部分

Controller

接收用户请求，委托给模型处理，并将处理完毕的数据返回给视图，再由视图负责展示，充当调度员的工作。

也就是GRASP中的Controller，授权其它类对应的职责，协调控制其对应的行为，自己不处理过多事务。

模型演变

Model1时代

在最开始，仅分为两层：视图层与模型层，视图层与控制层均由jsp实现，jsp具备显示和中间调度的双重职责。

  • 优点：架构简单、开发快、适合小型网站
  • 缺点：jsp职责不单一、难以维护

Model2时代

到了第二代，就是现在常用的模型、视图、控制器架构，前端展示与控制器的职责分离。

以典型的jsp+servlet+javabean举例：

  • 用户发请求，jsp接收参数并传递给servlet
  • Servlet接收请求数据，并调用对应dao与service的业务逻辑方法
  • 业务处理完毕，返回更新后的数据给servlet
  • servlet转向到JSP，由JSP来生成页面
  • 响应给前端更新后的页面

MVC框架的优缺点

优点

耦合性低

  • 视图层和业务层分离，这样就允许独立改变模型层或者视图层的代码而不对其他层产生影响
  • 模型是自包含的，与控制器和视图相分离，很容易改变应用程序的数据层和业务规则，如更换数据库。
  • MVC的应用程序的三个部件相互独立，所以依据这种设计思想能构造良好的低耦合的构件。

重用性高

  • MVC模式允许使用各种不同视图来访问同一个服务器端的代码，多个视图能共享一个模型，例如尽管移动端与pc端的界面不同，背后处理业务的逻辑是共同的，由此就省去了大量的开发成本。
  • 由于模型返回的数据没有进行格式化，所以同样的构件能被不同的界面使用。HTML与WAP格式的数据，仅仅需要改变视图层的实现方式，而控制层和模型层无需做任何改变。

生命周期成本低

  • 由于良好的分层、低耦合、高度的抽象，使得开发和维护用户接口的技术含量降低，方便快速迭代

开发快、可维护性高

  • 使用MVC模式使开发时间得到相当大的缩减，前后端各司其职、职责分离，方便进行项目的修改与维护。

缺点

没有明确的定义，抽象困难

  • 内部原理复杂，模型、控制器和视图要严格的分离，为架构与调试工作带来了困难。

一定程度上影响系统性能

  • 严格遵循MVC，使模型、视图与控制器分离，层数越多开销越大，降低运行效率。
  • 依据模型操作接口的不同，视图可能需要多次调用才能获得足够的显示数据，损害操作性能。

MVP

MVP其实是MVC的封装和演化，具体相比于MVC的变化有：

  • Controller被拆分，只用它处理View的点击事件，数据绑定，等处理
  • View被拆分，更加专注于视图的更新，只做跟视图相关的操作
  • Presenter被独立出来，用于沟通View和Model之间的联系

与MVC相比，MVP里Model不能直接作用于View 的更新，只能通过Presenter来通知View进行视图的刷新，避免了View 有过多的逻辑处理，更加简单。而Presenter持有了Model。Model 只用于处理跟数据获取相关的逻辑。

不同于MVC的model数据更新之后由model更新view数据，MVP中的presenter和View双向绑定，presenter对model单向绑定。更新只能由presenter传给view，调用一次方法才能更新数据，各层之间更加独立。

从而具有以下优点：

  • 模型与视图更彻底的分离，我们可以修改视图而不影响模型
  • 所有的交互都发生在Presenter内部，其他类的独立性加强，更加满足低耦合的要求。
  • 可以将一个Presenter用于多个视图，不需要改变Presenter的逻辑。
  • 如果我们把逻辑放在Presenter中，方便进行单元测试

美中不足的是由于view与presenter绑定很紧，很多渲染的工作嫁给了presenter完成。在修改视图代码时presenter代码很可能需要调整。同时presenter与view频繁交互，降低效率。

MVVM

WPF

WPF是微软新一代图形系统，运行在.NET Framework 3.0及以上版本下，为用户界面、2D/3D 图形、文档和媒体提供了统一的描述和操作方法。基于DirectX 9/10技术的WPF不仅带来了前所未有的3D界面，而且其图形向量渲染引擎也大大改进了传统的2D界面。

程序员在WPF的帮助下，要开发出媲美Mac程序的酷炫界面已不再是遥不可及的奢望。 WPF相对于Windows客户端的开发来说，向前跨出了巨大的一步，它提供了超丰富的.NET UI 框架，集成了矢量图形，丰富的流动文字支持(flow text support)，3D视觉效果和强大无比的控件模型框架。

MVVM概述

  • MVVM（Model-View-ViewModel），本质上就是MVC 的改进版。MVVM 将View 的状态和行为抽象化，将视图 UI 和业务逻辑分开。 ViewModel可以取出 Model 的数据同时帮忙处理 View 中由于需要展示内容而涉及的业务逻辑。微软的WPF带来了新的技术体验，促进了软件UI层更加细节化、可定制化，同时WPF也带来了 诸如Binding、Dependency Property等新特性。
  • MVVM框架的由来便是MVP（Model-View-Presenter）模式与WPF结合的应用方式时发展演变过来的一种新型架构框架，立足于原有MVP框架并且把WPF的新特性糅合进去，以应对客户日益复杂的需求变化。

分层分析

view与viewmodel双向绑定，viewmodel与model之间也是通过不同的格式互相传输数据。

View

View 是视图层，也就是用户界面。前端主要由 HTML 和 CSS 来构建，仅负责简单的数据展示，甚至渲染部分也是交给了viewmodel，复杂的前端展示如js，被划分到了紧密相关的为其定制的viewmodel层。

Model

Model 是指数据模型，泛指后端进行的各种业务逻辑处理和数据操控，这里与MVC、MVP区别不大

Viewmodel

  • ViewModel 是由前端开发人员组织生成和维护的视图数据层。
  • 在这一层，前端开发者对从后端获取的数据进行二次封装，以生成符合 View 层使用预期的视图数据模型。 ViewModel 所封装出来的数据模型包括视图的状态和行为两部分，而 Model 层的数据模型是只包含状态的，各区域具体的展示以及事件的响应都会对应在ViewModel完成封装。这样的封装使得 ViewModel 可以完整地去描述 View 层。
  • 由于实现了双向绑定，ViewModel 的内容会实时展现在 View 层，不必低效又麻烦地通过操纵 DOM 去更新视图，开发者只需要处理和维护 ViewModel，更新数据视图就会自动得到相应更新，真正实现数据驱动开发。
  • View 层展现的不是 Model 层的数据，而是 ViewModel 的数据。由 ViewModel 负责与 Model 层交互，这就完全解耦了 View 层和 Model 层，它是前后端分离方案实施的重要一环。

MVVM优点

  • 可重用，你可以把一些视图逻辑放在一个ViewModel里面，让很多view重用这段视图逻辑。
  • 独立开发，开发人员可以专注于业务逻辑和数据的开发（ViewModel），设计人员可以专注于页面设计，前后端分离
  • 可测试，viewmodel与view紧密链接，实时同步，测试可以针对ViewModel来写，大大降低测试难度。

总结

  • 我认为三种模式共有的目的就是实现model与view的完全分离，在MVC阶段，model的更新对view的更新还相对直接。到了MVP阶段，通过presenter对view以调用方法而非直接传数据的更新方式的改变实现了进一步的分离。而在MVVM中viewmodel对view的定制以及对model的二次封装实现了更加纯粹的分离
  • MVC的结构相对比较简单，基本可以满足model与view的分离。但是随着前端界面的发展，对前端的二次渲染与后端数据的再封装提出了更高的要求，进而有了MVP并最终过渡到了MVVM，实现了前端与后端的完全分离
  • 相应的对规范也提出了更高的要求，在MVC是很多时候servlet、html、数据库、业务逻辑都是需要一定的了解，而MVVM前后端完全分离后，对后端处理的数据格式有了更高的要求，以保证前端和后端的协作顺畅
  • 自然，对数据的测试重要性也愈发明显，后端业务逻辑需要测试、数据格式需要测试、层层之间的链接需要测试、前端发布需要测试。不过同样的也是测试技术的进步，前端可以在独立于后端的情况下仅使用虚拟的viewmodel层便可以完成测试，加快了测试进度
  • 层层之间的分离越发严谨，自然也保证了各层之间的安全，不可以跨层调用方法
  • 同时viewmodel的高度综合以及各层之间日益严谨的规范使得可复用性大大增强，开发的周期与成本降低
  • 虽然架构可以方便大型项目的迭代管理，但是同样意味着更低的性能与更高的初期开发成本，小型项目应该选用合适的架构。
