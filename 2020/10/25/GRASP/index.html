GRASP

发表于 2020-10-25 分类于 设计模式

概述

GRASP(General Responsibility Assignment Software Patterns )，即通用职责分配软件模式

通常在面向对象设计中指导为类和对象分配职责

GRASP中概念主要有:

  • Information Expert
  • Creator
  • Controller
  • Low Coupling
  • High Cohesion
  • Polymorphism
  • Pure Fabrication
  • Indirection,
  • Protected Variations

这些模式解决了常见的许多软件设计问题。这些技术的发明虽然没有创造新的工作方法，但是为面向对象设计中经过测试检验的已经存在的准则，提供了更好的记录和规范化途径。

Information Expert

信息专家决定如何分配职责，即对于给定的职责，为其分配所需的信息，决定信息储存在何处。

  • 某个类拥有完成某个职责所需要的所有信息，这个职责就应该分配给这个类来实现
  • 专家模式用来分配职责，根据给定的职责去查询和存储信息

由此，设想一个场景，市场上有一些商品，储存着与商品相关的所有信息。

要求实现判断某一商品是不是给定的一个商品种类（如苹果是不是一种桃子）

那么根据信息专家原则，该职责应该分配给商品本身。

Creator

通俗的来说，B是A的Creator，当且仅当满足以下一个或多个条件：

  • B的实例是A的实例的容器
  • B的实例是A的实例的聚合
  • B的实例记录A的实例
  • B的实例密切地使用A的实例
  • B的实例含有A的实例的初始化信息并传递给构造方法

Low Coupling（低耦合）

低耦合是一个估计单元，其规定了分配职责的准则：

  • 类间依赖性低
  • 类的改变对其他类影响小
  • 高可复用可能性

下面情况会造成A、B间的耦合：

  • A是B的属性
  • A调用B实例中的方法
  • A方法中引用了B
  • A实现B

具体来说，要避免下面的情况：

  • Don’t Talk to Strangers ，意思就是说，不需要通信的两个对象之间，不要进行无谓的连接。
  • 如果 A 已经和 B 有连接，如果分配 A 的职责给 B 不合适的话（违反信息专家模式），那么就把 B 的职责分配给 A。
  • 两个不同模块的内部类之间不能连接。
一和三比较容易理解，具体考虑一下第二项：

假设管理员管理着订单，而他想要统计订单数量，那么显然这个职责分配给管理员是不合适的，违反信息专家原则。

那么应该把职责分配给订单，提高订单的内聚性，降低类间耦合。

Controller

  • 控制器对象负责接收处理系统事件，应用情境的控制器应该用来处理情境中所有的系统事件。
  • 它是除了前端之外第一个接收协调控制系统操作的对象。
  • 控制器应该授权其它类对应的职责，协调控制其对应的行为。自己不应处理过多事务

High Cohesion（高内聚）

  • 高内聚致力于保证对象间的高聚集性、易管理型、可理解性，为类间低耦合提供支持
  • 高聚合意味着一组元素是高度相关且高度聚合的。将一个大型系统分解为具体类和子系统可以增加系统内聚性。否则，过于冗杂的元素会使系统聚合性降低。
  • 低聚合会导致难以理解、难以复用、难以维护、难以修改

专职由专人来做，与超市订单相关的如统计订单信息、修改某一商品信息、删除商品信息，应该由订单类完成

相对的，如订单不直接相关，比如维修与进货事件，应该交给其他的人来做。

Pure Fabrication

  • 纯虚构不在问题领域中代表任意概念，仅仅为实现类间低耦合、类内高聚合、可复用而存在
  • 在领域驱动设计中一般称为Service

顶层的Iterable与Collection抽象允许java对可迭代与收集类进行自由的扩展，按照Collection官方文档所说：

The JDK does not provide any direct implementations of this interface: it provides implementations of more specific subinterfaces like Set and List. This interface is typically used to pass collections around and manipulate them where maximum generality is desired.

官方没有提供直接的实现，而是提供了子接口Set和List。所以Colection用于最顶层的数组与集合传递。

由此便实现了低耦合、高内聚、可复用、以及高扩展性。

Indirection

  • 间接模块用于支持低耦合，通常通过分配为直接对象分配调节实现。
  • 常见的例子是在model和data层之间加入控制器组件

我认为控制器就是最典型的间接层

Protected Variations

  • 保护元素不受其他元素的变化的影响
  • 可以通过将不确定的模块用抽象类封装，通过多态实现接口实现

还是商品类的例子，尽管具体的商品千变万化，可以是苹果可以是梨。价格和具体种类也不定。

但是把商品的共性提取成父类再设计接口掩盖底层细节，便可借助多态可以实现保护元素的不变性。
